클라우드 네이티브 준비 기준

클라우드 네이티브 ISMS-P 준비는 범위와 증적 기준을 먼저 맞추는 일입니다.

AWS, EKS/ECS, SaaS, CI/CD, 개인정보 처리 시스템이 섞인 환경에서는 심사자가 묻는 기준과 실제 운영 데이터가 쉽게 어긋납니다.

KISA 기준 구조

ISMS-P 요구사항은 16/64/21 구조로 나뉩니다.

항목 수를 외우는 것보다 인증범위, 개인정보 흐름, 원본 증적의 기준을 같은 언어로 설명하는 일이 더 중요합니다.

16관리체계 수립 및 운영
64보호대책 요구사항
21개인정보 처리단계별 요구사항
scope

인증범위

서비스, AWS 계정, 리전, VPC, EKS/ECS, DB, SaaS, 수탁사 경계를 먼저 고정합니다.

evidence

원본 증적

정책 문서만으로는 부족합니다. 로그, 승인, 티켓, 배포, 백업 실행 이력이 함께 필요합니다.

privacy

개인정보 흐름

수집, 이용, 보관, 제공, 파기 흐름이 코드와 데이터 저장소 변경에 맞춰 갱신되어야 합니다.

어려운 이유

클라우드 네이티브 환경에서는 심사 질문과 운영 현실이 쉽게 어긋납니다.

리소스와 권한, 로그, 개인정보 흐름이 계속 바뀌기 때문에 한 번 만든 표만으로는 설명이 어렵습니다.

01

동적 AWS 리소스

계정, 리전, VPC, EKS 클러스터, 데이터 저장소가 계속 생기고 사라집니다.

02

흩어진 권한

IAM, SSO, EKS RBAC, DB 권한, SaaS 권한을 같은 사람 기준으로 봐야 합니다.

03

분산된 로그

CloudTrail, EKS audit, 애플리케이션 로그, DB 감사 로그가 사건 흐름으로 이어져야 합니다.

04

코드와 묶인 개인정보 증적

개인정보 수집 목적, 처리 위치, 보유 기간, 파기 내역이 서비스 변경과 함께 추적되어야 합니다.

05

외부 SaaS와 원격근무

SaaS, 수탁사, 단말, 원격 접속 경계가 넓어지면 제품명보다 책임 경계가 먼저 필요합니다.

P0 심사 리스크

심사 전에 반드시 닫아야 할 P0 리스크입니다.

P0는 제품 도입 여부와 관계없이 먼저 증명해야 하는 항목입니다. 이 공백은 준비 단계에서 집중해서 확인합니다.

P0

인증범위·자산·개인정보 흐름 누락

서비스, 계정, 데이터 저장소, 외부 연동이 목록과 다르면 이후 증적도 흔들립니다.

scope gap
P0

퇴사자/직무변경자 접근권한 잔존

SSO, IAM, DB, 쿠버네티스 권한 회수 내역을 사람 기준으로 보여줘야 합니다.

access review
P0

운영망과 개인정보처리시스템 접근 경로 불명확

누가 어떤 경로로 운영 시스템에 접속하는지 세션과 승인 흐름이 필요합니다.

admin path
P0

로그 수집·보존·무결성 부족

탐지 로그와 별도로 원본 로그의 보존 기간, 변경 불가, 암호화가 설명되어야 합니다.

log integrity
P0

취약점 진단과 조치 추적 미흡

스캔 결과보다 더 중요한 것은 담당자, 조치 기한, 재검증 내역입니다.

remediation
P0

파기 자동화와 증적 부재

보유 기간이 지난 개인정보가 실제로 삭제됐는지 시스템 로그로 남겨야 합니다.

retention
P0

사고대응/백업복구 증적 부족

훈련, 보고, 백업, 복구 테스트가 문서가 아니라 실행 이력으로 남아야 합니다.

response
